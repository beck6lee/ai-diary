# ADR-001 移动端 H5 优先，而非原生 App

## 背景

产品处于 0→1 阶段，核心假设尚未验证（AI 整理能否真正降低记录门槛）。需要选择初期发布形态。

## 备选方案

### 方案 A：移动端 H5（已选择）

**优点：**
- 开发周期短，一套代码跑全平台
- 无需 App Store 审核，随时发布/更新
- GitHub Pages 可免费托管，0 运营成本
- 用 URL 即可分享给用户试用

**缺点：**
- 无法发推送通知（无法提醒用户记日记）
- 不能调用相机、相册等原生能力（MVP 暂不需要）
- 浏览器存储有被清除的风险

### 方案 B：React Native / Flutter 原生 App（已放弃）

**优点：**
- 推送通知、相机、离线体验更好
- 可上架 App Store 触达更多用户

**缺点：**
- 开发周期长 2-3 倍
- App Store 审核周期不可控
- 两端（iOS + Android）维护成本高
- 验证阶段浪费资源

## 最终决策及理由

**选择方案 A（H5）**。

MVP 阶段最重要的事是用最低成本验证核心假设。H5 的发布速度和零成本部署远比原生体验重要。如果产品方向验证成立，再基于 H5 代码套壳（Capacitor / WebView）或重写原生版本。

## 影响与后续

- 用户数据存储在 localStorage，存在被浏览器清除的风险——这是已知的、可接受的 MVP 代价
- 日记提醒功能（帮用户养成习惯）在 H5 上难以实现，Phase 2 需要重新评估

## 回顾时机

当出现以下情况时，重新评估此决策：
- 月活用户超过 1000，且有明显的"存储丢失"投诉
- 产品验证成功，准备进入增长期
