# RFC-008 后台日记 Agent

## 背景

当前前端直接调用 DeepSeek API，每次用户新增一条记录时，都将当天**全部原始内容**重新发给 LLM，要求其输出完整日记。这带来几个问题：

1. **体验差**：随着当天记录增多，每次整理的时间越来越长（要重新生成越来越多的内容）
2. **不稳定**：全量重写时，AI 可能改动已经整理好的早期条目，导致内容漂移
3. **浪费**：大量 Token 花在重新处理已经处理过的内容上
4. **能力受限**：纯 prompt 难以实现记忆、工具调用、多轮推理等 Agent 能力

## 核心思路

引入后台服务，实现**增量更新**：

- Agent 维护当前日记的结构化状态
- 每次只处理**新增的一条**原始记录
- 决定将其插入哪个章节（今日工作/想法/知识碎片/复盘），或新开章节
- 返回更新后的完整日记

前端不再直接调用 DeepSeek，改为调用自建后端 API。

## 待定问题（设计阶段解决）

- 后端语言和框架：Node.js / Python？
- Agent 框架：原生实现 / LangChain / 其他？
- 部署方案：Railway / Fly.io / Vercel Functions / 自己的服务器？
- 用户鉴权：如何区分不同用户？前端的 API Key 是否仍由用户自持？
- 数据是否迁移到服务端存储，还是仍保留 localStorage？
- 流式输出是否继续支持？

## 预期收益

- 每次整理只处理新增内容，速度显著提升
- 已整理内容不会被意外改写
- 后端可扩展：记忆、搜索、目标追踪等能力更容易集成
- API Key 不再暴露在前端（可由服务端统一管理）
